# 飞书企业级长程协作记忆系统

## 白皮书 v1.0

---

## 1 引言

### 1.1 问题背景

现代企业中，跨部门协作的核心载体已从邮件和会议纪要迁移至即时通讯工具。飞书群聊承载了大量技术决策、待办分配、项目规范讨论和疑问解答。然而，这些高价值信息呈现出三个突出特征：**高度碎片化**、**随时间漂移**、**跨群扩散**。当团队成员需要回溯某项决策的来龙去脉时，往往面临"信息大海捞针"的困境。

更严重的是，现有 AI 助手在处理协作信息时普遍存在**"失忆症"**：每次交互仅基于当前对话窗口，无法维持跨会话的持续记忆。一个典型场景是——周一的架构决策到了周五的群聊中已被遗忘，团队成员可能做出与既有决策相矛盾的判断。

### 1.2 现有方案的不足

| 方案 | 局限 |
|------|------|
| 聊天搜索 | 关键词匹配，无法理解语义；结果噪音极大；无法区分已失效决策 |
| 简单 RAG | 将所有消息平等对待，缺乏时间有效性模型；无冲突解决机制 |
| 知识库 Wiki | 依赖人工维护，实时性差；无法自动追踪决策演化 |
| 对话历史摘要 | 信息压缩过度；无法区分决策/待办/结论/疑问的不同生命周期 |

### 1.3 本系统的目标

我们提出**飞书企业级长程协作记忆系统**，旨在实现：

1. **全量保留**：原始消息不可变、不丢失，作为信任基础
2. **结构提炼**：从碎片化对话中自动提取结构化事件，区分四类信息（决策/待办/结论/疑问）及其不同生命周期
3. **知识固化**：将跨时间的事件合并为稳定的组织知识，支持决策演化的版本追踪
4. **智能检索**：三层并行检索，自动解决信息冲突，确保回答的时效性和准确性

## 2 理论基础

### 2.1 认知科学记忆模型映射

人类记忆系统由 Atkinson-Shiffrin 模型描述为三阶段加工：感觉记忆（Sensory Memory）、短期记忆（Short-term Memory）和长期记忆（Long-term Memory）。本系统将这一认知架构映射为企业协作信息的处理流程：

```
认知模型                    系统架构
─────────                  ─────────
感觉记忆 ──────────────────→ Layer 1: Raw (原始消息)
  (瞬时的、未加工的)           (完整保留、不可变、仅追加)

短期记忆 ──────────────────→ Layer 2: Episodes (结构化事件)
  (经注意选择、暂时保持)       (四标签分类、MEM0四操作、时间有效性)

长期记忆 ──────────────────→ Layer 3: Knowledge (固化知识)
  (经复述与整合、持久存储)     (四类知识文件、跨时间稳定、状态追踪)
```

映射的形式化定义：

$$\text{Raw} = \{(m_1, m_2, \ldots, m_n) \mid m_i \in \mathcal{M}\}$$

$$\text{Episodes} = \{e \mid e = f_{\text{extract}}(\text{Raw}_{\Delta t}),\ e.\text{tags} \in \{\text{决策}, \text{待办}, \text{结论}, \text{疑问}\}\}$$

$$\text{Knowledge} = g_{\text{consolidate}}(\text{Episodes}_{\text{active}})$$

其中 $f_{\text{extract}}$ 为 LLM 驱动的事件提取函数，$g_{\text{consolidate}}$ 为知识合并函数。

### 2.2 MEM0 记忆操作理论

借鉴 MEM0 的记忆管理范式，本系统在 Episode 层实现四种操作（OpType）：

| 操作 | 形式化定义 | 语义 |
|------|-----------|------|
| **ADD** | $\text{ADD}(e) \Rightarrow e.\text{valid\_until} = \text{None}$ | 全新事件，现有记录中无同主题 |
| **UPDATE** | $\text{UPDATE}(e, e_{\text{old}}) \Rightarrow e_{\text{old}}.\text{valid\_until} = t_{\text{now}},\ e.\text{supersedes} = e_{\text{old}}.\text{ep\_id}$ | 已有事件的状态/内容更新 |
| **DELETE** | $\text{DELETE}(e_{\text{old}}) \Rightarrow e_{\text{old}}.\text{valid\_until} = t_{\text{now}}$ | 被否决的旧决策，标记失效 |
| **NOOP** | $\text{NOOP}(e) \Rightarrow \emptyset$ | 重复信息，跳过不写入 |

核心不变量：**任何 Episode 一旦创建，其内容不可修改，仅通过 valid_until 字段标记其有效性状态。** 这确保了信息演化的完整可追溯性。

操作决策函数由 LLM 在提取阶段确定，同时辅以基于标题相似度的快速预判机制：

$$\text{Op}(e) = \begin{cases} \text{UPDATE} & \text{if } \exists\, e' \in \text{Episodes}_{\text{active}} : \text{sim}(\text{title}(e), \text{title}(e')) \geq \theta \\ \text{ADD} & \text{otherwise} \end{cases}$$

其中 $\theta$ 为相似度阈值（默认 0.6），$\text{sim}$ 采用 Jaccard 词重叠率。

### 2.3 企业协作信息流特性

企业协作信息流具有与普通聊天截然不同的四个特性，它们直接决定了系统设计：

1. **决策的时效性**（Temporal Validity）：一项技术决策在特定时间窗口内有效，但可能被后续决策推翻。系统必须维护 `valid_from` / `valid_until` 时间窗口模型。

2. **待办的状态性**（Statefulness）：待办事项具有"开放→完成"或"开放→取消"的状态转换，而非简单存在或不存在。

3. **结论的稳定性**（Stability）：共识性结论一旦确立，变化频率远低于决策。适合作为知识层的高密度内容。

4. **疑问的开放性**（Openness）：疑问从提出到解答可能跨越数天甚至数周，需要持续追踪直到关闭。

## 3 架构设计

### 3.1 系统总览：三Agent三层架构

```
┌─────────────────────────────────────────────────────────┐
│                    飞书群聊 / 文档 / 日历                  │
└───────────────────────┬─────────────────────────────────┘
                        │ 采集
          ┌─────────────▼─────────────┐
          │   Collector Agent         │
          │   「记忆采集员」            │
          └─────────────┬─────────────┘
                        │ 仅追加写入
          ┌─────────────▼─────────────┐
          │  Layer 1: Raw 原文存档     │  ← 不可变信任基础
          │  (JSONL, 按日期分目录)     │
          └─────────────┬─────────────┘
                        │ LLM 提取
          ┌─────────────▼─────────────┐
          │   Memory Agent            │
          │   「记忆中枢」              │
          └──────┬──────────────┬─────┘
                 │              │
          ┌──────▼──────┐ ┌────▼──────────┐
          │ Layer 2:    │ │ Layer 3:      │
          │ Episodes    │→│ Knowledge     │
          │ 结构化事件   │ │ 固化知识       │
          └──────┬──────┘ └────┬──────────┘
                 │              │
          ┌──────▼──────────────▼─────┐
          │   Assistant Agent         │
          │   「项目助理」              │
          │   (三层并行只读检索)        │
          └──────────────────────────┘
```

### 3.2 Layer 1：Raw 原文存档

**设计原则：不可变的信任基础（Trust Foundation）**

Raw 层是整个系统的信息源头和审计基石。其核心约束为：

- **仅追加写入（Append-Only）**：消息一旦写入，永不修改、永不删除
- **原始格式保留**：完整保存发送人、时间戳、消息类型、来源链接等元数据
- **按日期分片**：每天一个 JSONL 文件，路径格式 `YYYY/MM/DD.jsonl`

数据格式定义：

```python
Message {
    msg_id: str           # 全局唯一消息ID
    chat_id: str          # 所属群聊ID
    sender_id: str        # 发送人ID
    sender_name: str      # 发送人姓名
    content: str          # 消息原文
    msg_type: str         # 消息类型 (text/image/file...)
    timestamp: str        # ISO 8601 时间戳
    parent_id: str?       # 回复的消息ID
    source: Enum          # group_chat / doc / calendar / minutes
    source_url: str?      # 飞书消息链接 (可溯源)
    is_meeting_end: bool  # 会议结束标记 (触发器信号)
    collected_at: str     # 采集时间
}
```

索引策略：按日期文件组织，支持时间范围查询（`read_range`）和增量读取（`read_new_since`），无需额外索引即可满足条件触发器的消息计数需求。

### 3.3 Layer 2：Episodes 结构化事件

**设计原则：四标签分类 + MEM0 四操作 + 时间有效性**

Episode 层是从原始消息到结构化知识的关键中间层。

#### 3.3.1 四标签分类体系

| 标签 | 定义 | 生命周期特征 | 典型模式 |
|------|------|-------------|---------|
| **决策** | 团队达成的正式结论，有明确的采纳表述 | 可被后续决策推翻 | "我们决定采用 gRPC" |
| **待办** | 被分配了负责人的行动项 | 开放→完成/取消 | "张三负责本周完成接口文档" |
| **结论** | 共识性认知，不一定有行动 | 相对稳定 | "性能瓶颈在数据库连接池" |
| **疑问** | 明确提出但尚无答案的问题 | 开放→已解答 | "缓存策略选 Redis 还是 Memcached？" |

#### 3.3.2 时间有效性模型

每个 Episode 携带时间有效性窗口：

$$\text{Episode} = (e.\text{valid\_from},\ e.\text{valid\_until}) \quad \text{where } e.\text{valid\_until} \in \{\text{None},\ t_{\text{expire}}\}$$

- `valid_until = None`：当前有效（Active）
- `valid_until = t`：已于时间 $t$ 失效

判断函数：$e.\text{is\_active}() \iff e.\text{valid\_until} = \text{None}$

#### 3.3.3 血缘追踪（Lineage）

每个 Episode 通过 `source_msg_ids` 字段指向 Layer 1 的原始消息，通过 `supersedes` 字段指向被其替代的旧 Episode，形成双向可追溯链条：

$$\text{lineage}(e) = e.\text{source\_msg\_ids} \xrightarrow{L1} \text{Raw} \quad \cup \quad e.\text{supersedes} \xrightarrow{L2} \text{Episode}_{\text{old}}$$

这确保了任意一条结构化事件都可以回溯到原始对话上下文。

#### 3.3.4 Episode 数据结构

```python
Episode {
    ep_id: str              # EP-YYYYMMDD-NNN
    date: str               # YYYY-MM-DD
    title: str              # 事件标题 (≤20字)
    summary: str            # LLM 生成摘要 (50-100字)
    tags: list[str]         # TagType 值列表
    source_msg_ids: list    # 指向 L1 消息ID
    participants: list[str] # 参与人
    operation: str          # ADD/UPDATE/DELETE/NOOP
    supersedes: str?        # 被替代的旧 Episode ID
    valid_from: str         # ISO 8601 生效时间
    valid_until: str?       # None=当前有效
    confidence: float       # 提取置信度 [0,1]
    created_at: str         # 创建时间
}
```

### 3.4 Layer 3：Knowledge 固化知识

**设计原则：跨时间稳定 + 四类知识文件 + 议题状态机**

Knowledge 层是信息密度最高、稳定性最强的层，直接服务于 Assistant Agent 的检索与回答。

#### 3.4.1 四类知识文件

| 文件 | 内容 | 更新频率 | HEARTBEAT 关联 |
|------|------|---------|---------------|
| **技术决策库.md** | 所有正式技术决策，含决策原因和时间窗口 | 低频（仅当决策被推翻时更新） | 否 |
| **待跟进议题.md** | 待办和疑问类事件，含开放/已关闭状态 | 高频（议题状态变更时更新） | **是（主动推送数据源）** |
| **项目规范.md** | 团队约定的工作方式和偏好 | 极低频 | 否 |
| **人员职责.md** | 谁负责什么，联络方式 | 低频 | 否 |

#### 3.4.2 待跟进议题状态机

```
           新增待办/疑问
               │
               ▼
         ┌──────────┐
         │   开放    │ ←── 持续追踪
         └────┬─────┘
              │
     ┌────────┴────────┐
     │                 │
  已解决/完成       被否决/取消
     │                 │
     ▼                 ▼
┌──────────┐    ┌──────────┐
│  已关闭   │    │  已关闭   │
└──────────┘    └──────────┘
```

状态转换规则：当 Knowledge 合并引擎检测到新的决策（Decision 标签）解决了某个疑问，或新的确认完成了某个待办时，将对应议题从"开放"转为"已关闭"。

#### 3.4.3 Episode 到 Knowledge 的合并规则

$$\text{Knowledge}_{\text{new}} = g_{\text{consolidate}}(\text{Episodes}_{\text{new}},\ \text{Knowledge}_{\text{current}})$$

合并规则由 LLM 执行，但遵循以下确定性约束：

1. 已有的正确知识保留不动
2. 新知识追加到对应分类文件
3. 被否决/更新的旧知识标记 `失效时间`
4. 每条知识必须标注来源 Episode ID（`source_ep_ids`），维持血缘链

### 3.5 三层触发机制

信息的逐层提炼不是实时流式处理，而是基于条件触发器（Trigger Engine）的批量处理。触发条件的形式化表达：

**Episode 提取触发**：

$$T_{\text{episode}} = (|\Delta\text{Raw}| \geq \theta_{\text{msg}}) \lor \text{meeting\_end}$$

其中 $\theta_{\text{msg}} = 20$（默认阈值），$\Delta\text{Raw}$ 为自上次处理以来的新增消息数。

**Knowledge 合并触发**：

$$T_{\text{knowledge}} = (|\Delta\text{Episodes}_{\text{active}}| \geq \theta_{\text{ep}}) \lor \text{cron}(02\!:\!00)$$

其中 $\theta_{\text{ep}} = 10$（默认阈值），$\text{cron}(02\!:\!00)$ 为每日凌晨定时触发。

**触发器状态追踪**（TriggerState）维护以下关键变量：

| 变量 | 语义 |
|------|------|
| `last_collect_time` | 上次采集完成时间 |
| `last_episode_process_time` | 上次 Episode 提取完成时间 |
| `last_knowledge_merge_time` | 上次 Knowledge 合并完成时间 |
| `new_message_count_since_process` | 自上次提取以来的新增消息计数 |
| `meeting_end_detected` | 是否检测到会议结束标记 |

## 4 Agent 设计

### 4.1 Collector Agent（记忆采集员）

**职责**：从飞书群聊、文档、日历采集原始数据，写入 Raw 层。

#### 4.1.1 HEARTBEAT 机制

采集器以固定心跳间隔（默认 30 分钟）运行：

```
每30分钟:
  1. 读取上次采集时间戳 (last_collect_time)
  2. 调用飞书 API / Lark CLI 拉取增量消息
  3. 将消息追加写入 Raw 层 (append-only)
  4. 检测会议结束标记 (is_meeting_end)
  5. 更新触发器状态 (mark_collected)
```

#### 4.1.2 飞书集成：Lark CLI

真实模式下，通过 `lark-cli` 命令行工具拉取飞书数据：

```bash
lark-cli im messages list --chat-id {chat_id} --start-time {since}
```

输出解析为 Message 对象，关键转换包括：
- `message_id` → `msg_id`
- `sender.id/name` → `sender_id/sender_name`
- `body.content` → `content`
- `create_time` → `timestamp`

#### 4.1.3 采集协议与容错

- **增量同步**：基于时间戳的增量拉取，避免重复采集
- **错误隔离**：单个群聊采集失败不影响其他群聊
- **重试机制**：Lark CLI 调用失败时指数退避重试（最多3次）
- **会议标记检测**：自动识别"会议结束"、"散会"、"会议纪要"等标记词

### 4.2 Memory Agent（记忆中枢）

**职责**：将 Raw 层消息提炼为结构化 Episode，并合并到 Knowledge 层。

#### 4.2.1 两阶段处理流水线

```
Raw 消息 ──→ [阶段1: 事件提取] ──→ 候选 Episodes ──→ [阶段2: 操作决策] ──→ 最终 Episodes
              (LLM 提取)                              (LLM + 规则)
```

**阶段1：事件提取**

将一批 Raw 消息连同现有 Episode 列表一起提交给 LLM，Prompt 要求：

> 从飞书群聊对话中提取所有值得记录的事件，每个事件包含：title（≤20字）、summary（50-100字）、tags（决策/待办/结论/疑问）、participants、source_msg_ids、operation（ADD/UPDATE/DELETE/NOOP）、supersedes、confidence（0-1）

**阶段2：操作决策**

LLM 在提取时同步决定操作类型，辅以快速预判：

```python
# 基于标题相似度的快速预判
if title_exact_match(existing_episode):
    operation = UPDATE
elif jaccard_overlap(title, existing_episode.title) >= 0.6:
    operation = UPDATE
else:
    operation = ADD
```

#### 4.2.2 幂等性保证

系统通过以下机制确保重复触发不会产生冗余数据：

1. **触发器状态追踪**：`last_episode_process_time` 确保每次只处理增量消息
2. **NOOP 操作**：LLM 可识别重复信息并返回 NOOP，跳过不写入
3. **supersedes 链**：UPDATE 操作明确指向被替代的旧 Episode，避免同一事件存在多个活跃版本
4. **置信度过滤**：`confidence < 0.5` 的提取结果被丢弃

### 4.3 Assistant Agent（项目助理）

**职责**：检索三层记忆，回答用户问题，主动推送待跟进议题。

#### 4.3.1 并行检索策略

用户提问后，同时向三层发出检索请求（详见第5节）。

#### 4.3.2 四种交互模式

| 模式 | 触发条件 | 处理流程 | 输出格式 |
|------|---------|---------|---------|
| **被动问答** | 被 @提及 或私聊 | 并行检索 → 排序合并 → LLM 生成 | 答案 + 来源标注(L1/L2/L3) |
| **主动推送** | 每日 10:00 定时扫描 | 读取开放议题 → 冷却过滤 → 格式化 | 议题 + 负责人 + 截止日期 |
| **定期摘要** | 日报 18:00 / 周报 周一 09:00 | 汇总 Episode → LLM 生成 | 关键决策/新增待办/已完成/待解决 |
| **决策追踪** | 命令触发 | 遍历 L2 决策 + L3 技术决策库 | 按时间排列的决策时间线 |

#### 4.3.3 主动推送逻辑

```python
每日 10:00:
  1. 读取 待跟进议题.md 中 status == "开放" 的条目
  2. 过滤: 距上次推送 > 24小时 (冷却期)
  3. 生成推送消息: 议题标题 + 负责人 + 截止日期 + 当前状态
  4. 通过飞书机器人发送
  5. 记录推送历史 (push_history)
```

冷却机制确保同一议题不会在短时间内重复骚扰团队成员。

## 5 并行检索策略

### 5.1 三层并发搜索

用户查询到达后，系统使用 `asyncio.gather` 同时向三层发起检索请求：

```python
l1_results, l2_results, l3_results = await asyncio.gather(
    search_raw(query, top_k),
    search_episodes(query, top_k),
    search_knowledge(query, top_k),
)
```

这种并发设计将检索延迟从 $t_{L1} + t_{L2} + t_{L3}$ 降低至 $\max(t_{L1}, t_{L2}, t_{L3})$。

### 5.2 优先级排序

检索结果按**层权重 x 相关性分数**排序：

$$\text{score}(r) = w_{r.\text{layer}} \times \text{relevance}(r)$$

层权重默认值：

| 层级 | 权重 $w$ | 理由 |
|------|---------|------|
| L3 Knowledge | 1.0 | 信息密度最高，经人工验证级别的提炼 |
| L2 Episodes | 0.7 | 结构化但可能包含已失效信息 |
| L1 Raw | 0.4 | 原始噪声大，仅用于溯源验证 |

排序规则（按优先级递减）：

1. $\text{score}(r)$ 降序
2. 同分时 $r.\text{layer}$ 降序（L3 > L2 > L1）
3. 同层同分时 $r.\text{timestamp}$ 降序（时间新的优先）

### 5.3 冲突解决规则

当不同层级返回了关于同一主题的矛盾信息时：

| 规则 | 形式化描述 |
|------|-----------|
| **L3 > L2 > L1** | $\forall\, r_3 \in L3,\ r_2 \in L2:\ r_3.\text{layer} > r_2.\text{layer} \Rightarrow r_3 \text{ wins}$ |
| **valid_until=null 优先** | $r.\text{valid\_until} = \text{None} \succ r.\text{valid\_until} \neq \text{None}$ |
| **UPDATE > superseded** | 新记录优先于被替代的旧记录 |
| **冲突标记** | 被淘汰的结果保留但分数乘以 0.3，标记 `conflict_superseded = True` |

### 5.4 结果融合算法

```
1. 三层并行搜索 → L1_results + L2_results + L3_results
2. 按 score = layer_weight × relevance 排序
3. 按主题分组 (标题关键词相似度)
4. 同组内检测冲突 (不同层级 + 内容差异)
5. 冲突解决: 保留胜出者，降权败者
6. 返回 top_k × 2 条结果 (供上层 LLM 选用)
```

最终融合结果同时包含高密度的 Knowledge 答案、上下文补充的 Episode 细节和可溯源的 Raw 原文，为 LLM 生成回答提供多粒度信息支撑。

## 6 实现与部署

### 6.1 技术栈

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| 后端框架 | Python / FastAPI | 异步原生，适合 LLM 调用和并发检索 |
| 定时调度 | APScheduler | 轻量级 cron + interval 调度 |
| LLM 接入 | OpenAI-compatible API | 支持 GPT-4o / DeepSeek / 本地 vLLM |
| 数据采集 | Lark CLI | 飞书官方命令行工具 |
| 存储层 | 文件系统 (JSONL + JSON + Markdown) | 零依赖、可审计、git 友好 |
| 异步并发 | asyncio + httpx | 高效的 I/O 并发 |

### 6.2 OpenClaw Workspace 配置

系统基于 OpenClaw 多 Agent 框架部署，每个 Agent 拥有独立的工作空间和四个核心配置文件：

**SOUL.md** — Agent 身份与核心原则

- Collector: "严谨的数据采集专员，完整采集、原始保留、守时可靠"
- Memory: "信息提炼专家，精准提炼、四标签分类、四操作写入"
- Assistant: "知识渊博的项目助理，准确回答、来源标注、主动推送"

**AGENTS.md** — Agent 间协作协议

```
数据流向:
  Collector  → write: [raw]
  Memory     → write: [episodes, knowledge], read: [raw, episodes]
  Assistant  → read: [raw, episodes, knowledge]
```

**HEARTBEAT.md** — 心跳与触发配置

- Collector: 每 30 分钟采集
- Memory: Episode 提取（>20条消息/会议结束）、Knowledge 合并（>10条Episode/每日02:00）
- Assistant: 推送扫描（每日10:00）、摘要生成（每日18:00）

**MEMORY.md** — Agent 状态追踪

记录各层的处理进度、计数器、知识层概况等运行时状态。

### 6.3 双模式架构

系统支持 Demo 模式和 Real 模式的无缝切换：

| 特性 | Demo 模式 | Real 模式 |
|------|----------|----------|
| LLM | MockLLMProvider (预定义输出) | OpenAIProvider (真实 API) |
| 采集器 | SimCollector (YAML 场景回放) | LarkCollector (飞书真实数据) |
| 触发方式 | API 手动触发 | APScheduler 自动调度 |
| 配置 | `mode: demo` | `mode: real` |

### 6.4 文件存储设计

```
memory_store/
├── raw/                        # Layer 1
│   └── 2026/04/16.jsonl       # 每天一个 JSONL 文件
├── episodes/                   # Layer 2
│   ├── _index.json            # Episode 索引 (快速查询)
│   ├── EP-20260416-001.json   # 每个 Episode 一个 JSON 文件
│   └── EP-20260416-002.json
└── knowledge/                  # Layer 3
    ├── _meta.json             # 合并元数据
    ├── 技术决策库.md           # 四类知识文件

```

选择文件系统而非数据库的核心理由：

1. **可审计性**：所有数据可直接用文本编辑器查看
2. **版本控制友好**：Markdown 和 JSON 天然适合 git 追踪
3. **零运维**：无需数据库服务，部署极简
4. **数据可移植**：标准格式，无厂商锁定

## 7 评测

### 7.1 八项评测指标

| 编号 | 指标名称 | 公式 | 评估对象 |
|------|---------|------|---------|
| M1 | Episode 提取准确性 | $F_1 = \frac{2PR}{P+R}$ | Raw → Episode 转换质量 |
| M2 | 标签分类准确性 | $\text{Acc}_{\text{tag}} = \frac{|\{e \mid \hat{T}(e) \cap T(e) \neq \emptyset\}|}{|E|}$ | 四标签分类正确性 |
| M3 | 操作正确性 | $\text{Acc}_{\text{op}} = \frac{|\{e \mid \hat{O}(e) = O(e)\}|}{|E|}$ | ADD/UPDATE/DELETE/NOOP 决策准确性 |
| M4 | 知识合并质量 | $\text{Coverage} = \frac{|\text{KW}_{\text{golden}} \cap \text{KW}_{\text{knowledge}}|}{|\text{KW}_{\text{golden}}|}$ | 关键词覆盖率 |
| M5 | 检索准确性 | $\text{MRR} = \frac{1}{|Q|}\sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}$ | 首个相关结果的排名倒数均值 |
| M6 | 冲突解决准确性 | $\text{Acc}_{\text{conflict}} = \frac{|\{q \mid \text{resolve}(q) = \text{expected}\}|}{|Q_{\text{conflict}}|}$ | 冲突场景下的正确率 |
| M7 | 主动推送时效性 | $\text{PushCov} = \frac{|I_{\text{pushed}} \cap I_{\text{open}}|}{|I_{\text{open}}|}$ | 开放议题的推送覆盖率 |
| M8 | 端到端延迟 | $P_{50},\ P_{95}$ of $\{t_{\text{query}} - t_{\text{response}}\}$ | 查询响应延迟分布 |

### 7.2 黄金标准数据集

评测依赖两个黄金标准文件：

- **golden_episodes.json**：标准 Episode 集合，含 `source_keywords`、`tags`、`operation` 等标注
- **golden_queries.json**：标准查询集合，含 `expected_answer_contains`、`should_resolve_conflict` 等标注

数据集应覆盖：
- 单标签/多标签 Episode
- 四种操作类型（重点覆盖 UPDATE 和 DELETE）
- 冲突场景（同一主题在不同层级有矛盾信息）
- 时间有效性场景（已失效决策不应出现在回答中）

### 7.3 消融实验设计

| 实验 | 移除组件 | 预期影响 |
|------|---------|---------|
| A1 | 移除 L3 Knowledge 层 | M4/M5 下降；冲突解决能力丧失 |
| A2 | 移除 MEM0 四操作（仅用 ADD） | M3 归零；冗余数据膨胀 |
| A3 | 移除时间有效性模型（valid_until） | 过期决策被错误引用 |
| A4 | 移除并行检索（串行） | M8 延迟增加 2-3 倍 |
| A5 | 移除冲突解决器 | M6 归零；矛盾信息直接呈现给用户 |
| A6 | 移除主动推送 | M7 归零；待办易被遗忘 |

## 8 相关工作

### 8.1 MEM0

MEM0 提出了大语言模型的记忆管理层，通过 ADD/UPDATE/DELETE/NOOP 四操作实现记忆的增量更新，避免简单追加导致的冗余和矛盾。本系统将 MEM0 的操作范式引入企业协作场景，并扩展了时间有效性模型（valid_from/valid_until）和血缘追踪（source_msg_ids），使其适应协作信息的时效性特征。

### 8.2 OpenClaw 原生记忆

OpenClaw 框架提供了 Agent 工作空间和 HEARTBEAT 机制，支持多 Agent 协作。本系统在 OpenClaw 的基础设施上构建了三层记忆架构，利用 SOUL.md/AGENTS.md/HEARTBEAT.md/MEMORY.md 四文件体系定义 Agent 行为，并将"待跟进议题.md"作为 HEARTBEAT 的数据源，实现了记忆系统与 Agent 框架的深度融合。

### 8.3 企业知识管理

传统企业知识管理（Confluence、Notion 等）依赖人工维护，存在实时性差、更新滞后的问题。本系统通过 LLM 驱动的自动提取和合并，实现了从"人写知识库"到"AI 提炼知识库"的范式转变，同时保留人类可编辑的 Markdown 格式，确保知识的可干预性。

### 8.4 RAG 系统

标准 RAG（Retrieval-Augmented Generation）将文档切片后进行向量检索，缺乏对信息时效性和冲突的处理能力。本系统的三层架构提供了信息密度的梯度（Raw → Episodes → Knowledge），并行检索策略兼顾了高密度答案和低层级溯源，冲突解决器确保了回答的一致性，这些是标准 RAG 所不具备的能力。

## 9 局限性与未来工作

### 9.1 向量语义搜索

当前系统的检索层基于关键词匹配，对语义相似但不包含查询关键词的内容召回能力有限。未来计划引入向量嵌入模型（如 text-embedding-3-small），在保留关键词检索作为快速通道的同时，增加语义检索路径，形成混合检索策略。

### 9.2 多项目支持

当前系统以单项目为边界组织记忆。在大型企业中，同一团队可能同时参与多个项目，不同项目间存在知识复用和决策关联的需求。未来将引入项目命名空间（Project Namespace）和跨项目知识引用机制。

### 9.3 跨群记忆共享

飞书中同一项目的信息散布在多个群聊中（核心群、技术群、测试群等）。当前采集器已支持多群监控，但跨群信息的去重和关联仍依赖 LLM 的隐式处理。未来将显式建模群间关系图，提升跨群信息融合的准确性。

### 9.4 记忆遗忘机制

当前系统仅通过 valid_until 标记信息的逻辑失效，不进行物理删除。随着系统长期运行，历史数据将持续膨胀。未来需要设计遗忘机制：
- 基于访问频率的衰减（近期频繁访问的记忆权重更高）
- 归档策略（超过保留期的 Raw 消息移至冷存储）
- 自动摘要压缩（将过细粒度的 Episode 压缩为更高层抽象）

## 10 结论

本文提出了飞书企业级长程协作记忆系统，通过三层记忆架构（Raw → Episodes → Knowledge）和三 Agent 协作机制（Collector → Memory → Assistant），解决了企业跨部门协作中的 AI "失忆症"问题。

系统的核心创新点包括：

1. **认知科学启发的三层架构**：将人类记忆的信息加工模型映射为企业协作信息的提炼流程，实现从碎片化消息到结构化知识的信息密度跃迁。

2. **MEM0 风格四操作引擎**：通过 ADD/UPDATE/DELETE/NOOP 操作和时间有效性模型（valid_from/valid_until），实现了协作信息的精确生命周期管理，避免了简单追加导致的冗余和矛盾。

3. **三层并行检索与冲突解决**：通过 asyncio.gather 并发检索三层记忆，按层权重排序，并自动检测和解决跨层信息冲突，确保回答的时效性和一致性。

4. **OpenClaw 深度集成**：基于 SOUL.md/AGENTS.md/HEARTBEAT.md/MEMORY.md 四文件体系定义 Agent 行为，将"待跟进议题.md"作为 HEARTBEAT 的数据源，实现记忆系统与 Agent 框架的有机融合。

5. **可审计的文件存储**：JSONL + JSON + Markdown 的文件存储方案，兼顾了机器可处理和人类可审计的双重需求，支持 git 版本追踪，部署零依赖。

系统已在 Demo 模式下完成端到端验证，涵盖数据采集、事件提取、知识合并、并行检索、主动推送和定期摘要的完整流程。评测框架覆盖八项指标，为系统优化提供了量化依据。随着向量语义搜索、多项目支持和记忆遗忘机制的引入，系统将进一步提升企业级协作记忆的智能水平。
